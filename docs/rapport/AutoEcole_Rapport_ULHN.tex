\section{Introduction et contexte}

\ProjectName{} est une application web de gestion d'auto-ecole concue autour du framework Symfony. L'application permet de suivre les candidats, les examens, les inscriptions, les paiements et les historiques administratifs. La version initiale couvrait deja les operations elementaires de gestion, mais elle restait limitee pour un usage demonstratif complet : environnement d'execution partiel, interface sommaire, migrations fragiles et logique metier dispersee dans plusieurs couches.

Le travail presente dans ce rapport consiste a transformer ce projet en une application plus robuste, plus facilement executable et plus proche d'un outil metier exploitable. L'objectif n'est pas seulement de dockeriser le depot ; il s'agit egalement de clarifier l'architecture, renforcer la logique de progression des examens, enrichir l'interface, ajouter des exports et preparer une base saine pour de futures extensions.

\begin{infobox}{Objectifs principaux}
\begin{compactitem}
  \item rendre l'application executable par Docker sans installation locale de PHP, Composer ou PostgreSQL ;
  \item fournir un script de lancement multiplateforme capable de detecter l'environnement ;
  \item nettoyer la configuration, les migrations et les secrets de developpement ;
  \item enrichir l'application avec des fonctions de pilotage, de filtrage, d'export et d'audit ;
  \item documenter le projet de maniere professionnelle et reproductible.
\end{compactitem}
\end{infobox}

\section{Analyse de l'existant}

Le projet initial etait organise comme une application Symfony classique avec des entites Doctrine, des controleurs CRUD et des templates Twig. Trois concepts principaux etaient deja presents : les candidats, les examens et les inscriptions de candidats aux examens.

\subsection{Points positifs}

\begin{compactitem}
  \item l'utilisation de Symfony fournit une base solide pour le routage, les formulaires, la validation et l'injection de dependances ;
  \item Doctrine structure la persistance autour d'entites explicites ;
  \item Dompdf etait deja present pour la generation PDF ;
  \item des tests metier existaient autour de la logique de paiement et de progression ;
  \item la separation minimale entre controleurs, entites, formulaires et repositories etait deja amorcee.
\end{compactitem}

\subsection{Limites identifiees}

\begin{warningbox}{Risques techniques de la version initiale}
\begin{compactitem}
  \item la configuration de base contenait une URL PostgreSQL locale et personnelle ;
  \item les migrations comportaient des traces d'un autre domaine metier et n'etaient pas propres pour une base vierge ;
  \item l'environnement Docker etait incomplet pour lancer l'application seule ;
  \item l'interface reposait sur un CDN externe et restait peu exploitable ;
  \item les listes ne proposaient pas de filtrage avance, de KPI ou de lecture decisionnelle ;
  \item certaines routes etaient semantiquement peu lisibles ;
  \item la logique metier devait etre davantage centralisee.
\end{compactitem}
\end{warningbox}

\section{Architecture cible}

L'architecture cible repose sur une separation claire entre couches. Les controleurs traitent les requetes HTTP, les repositories portent les requetes de consultation, les entites decrivent le modele persistant et le service \texttt{AutoEcoleManager} concentre les regles metier transverses.

\section{Conteneurisation et environnement d'execution}

La dockerisation s'appuie sur quatre services : l'application Symfony, PostgreSQL, Adminer et Mailpit. Le conteneur applicatif installe les extensions PHP necessaires, active Apache avec reecriture d'URL, installe Composer et applique les migrations au demarrage.

\begin{longtable}{p{3.2cm}p{4.2cm}p{7.1cm}}
\toprule
\textbf{Service} & \textbf{Image / role} & \textbf{Utilite} \\
\midrule
\endhead
\texttt{app} & PHP 8.3 Apache & Execute Symfony, installe les dependances Composer, applique les migrations et expose le port 8080. \\
\texttt{database} & PostgreSQL 16 Alpine & Stocke les candidats, examens et inscriptions. \\
\texttt{adminer} & Adminer & Permet d'inspecter la base depuis un navigateur. \\
\texttt{mailpit} & Mailpit & Prepare les evolutions de notification email sans serveur SMTP externe. \\
\bottomrule
\end{longtable}

\begin{lstlisting}[language=bash,caption={Lancement Docker principal}]
docker compose up --build
\end{lstlisting}

\subsection{Script multiplateforme}

Un script Python, \texttt{scripts/bootstrap\_environment.py}, detecte le systeme d'exploitation et verifie la presence de Docker ainsi que du plugin Docker Compose. Selon le systeme, il propose une commande adaptee : \texttt{winget} sous Windows, \texttt{brew} sous macOS, ou le gestionnaire de paquets disponible sous Linux.

\begin{decisionbox}{Choix de conception}
Le script reste volontairement prudent. Il n'installe pas Docker sans confirmation explicite. Cette approche evite d'executer des commandes systeme invasives sans accord de l'utilisateur, tout en gardant un flux simple pour un poste neuf.
\end{decisionbox}

\section{Regles metier}

La regle de progression des examens est centralisee. Un candidat peut passer le code en premiere etape. Le creneau devient accessible apres reussite du code. La conduite devient accessible apres reussite du creneau.

\begin{lstlisting}[language=PHP,caption={Principe de progression dans AutoEcoleManager}]
Code reussi -> Creneau autorise -> Conduite autorisee
\end{lstlisting}

Le paiement est egalement normalise. Un montant positif reduit le reste a payer. Le solde ne peut jamais devenir negatif. Lorsque le reste atteint zero, l'objet modifie automatiquement le statut financier concerne.

\section{Qualite logicielle et maintenabilite}

Le projet a ete retravaille pour limiter les dependances implicites. Les secrets personnels ont ete retires de la configuration versionnee, les migrations ont ete rationalisees et les styles sont servis localement. Les classes metier exposent des methodes calculees explicites : age, solde, progression paiement, progression examen et prochain examen.

\begin{infobox}{Points de maintenabilite}
\begin{compactitem}
  \item les controleurs restent responsables du flux HTTP ;
  \item les repositories concentrent les requetes de filtrage ;
  \item le service metier centralise les regles de progression ;
  \item les commandes CLI facilitent le seed et l'audit ;
  \item Docker fournit un environnement stable et partageable.
\end{compactitem}
\end{infobox}

\section{Validation et exploitation}

Une commande de controle locale est fournie :

\begin{lstlisting}[language=bash]
scripts/quality.sh
\end{lstlisting}

Dans un environnement Docker complet, cette commande peut etre completee par les lints Symfony, les tests PHPUnit et l'audit metier :

\begin{lstlisting}[language=bash]
docker compose exec app php bin/console lint:twig templates
docker compose exec app php bin/console lint:yaml config
docker compose exec app php bin/phpunit
docker compose exec app php bin/console app:quality-audit
\end{lstlisting}

\section{Limites et perspectives}

Cette version constitue une base solide pour une demonstration et une extension progressive. Elle ne couvre pas encore la gestion des moniteurs, les lecons de conduite, les contrats PDF signes, les pieces jointes, l'authentification par role ou l'envoi effectif de notifications. Ces evolutions sont naturelles et peuvent etre ajoutees sans remettre en cause l'architecture existante.

\begin{decisionbox}{Extensions prioritaires}
\begin{compactitem}
  \item authentification avec roles administrateur, secretaire et moniteur ;
  \item planning des heures de conduite ;
  \item generation de factures PDF ;
  \item depot et validation des pieces administratives ;
  \item notifications email via Messenger et Mailpit en developpement ;
  \item API REST pour une future interface mobile.
\end{compactitem}
\end{decisionbox}

\section{Conclusion}

Le projet \ProjectName{} a ete transforme en application Symfony dockerisee, mieux structuree et plus riche fonctionnellement. La mise en place de Docker rend l'execution reproductible, PostgreSQL remplace les dependances locales implicites, et l'interface fournit une lecture metier plus claire. Les ajouts en matiere de reporting, d'exports, d'audit et de controle de progression permettent de passer d'un CRUD simple a une application de gestion plus coherente et extensible.
